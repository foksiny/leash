Compiler & CLI Reference

The Leash compiler transforms .lsh source files into native executables through a 15-stage pipeline that spans lexical analysis, type checking, optimization, code generation via LLVM, and linking. You interact with this pipeline through the leash CLI, which provides commands for compilation, execution, static analysis, and project management.

Command Overview

CommandDescription
leash compile <file>Compile to a native executable
leash run <file>Run directly without creating a binary
leash dump <file>Dump LLVM IR to a .ll file
leash check <file>Static analysis without compiling
leash init [dir]Scaffold a new project
leash buildBuild project from config.lshc
leash runpCompile and run project from config.lshc
leash install <path>Install library to ~/.leash/libs/
leash updateUpdate Leash to the latest version

Leash can also compile programs into shared (.so) or static (.a) libraries for use by other programs or languages. This is useful for creating reusable components that can be linked into other projects.

Compilation Targets

Leash supports multiple compilation targets, allowing you to compile your code for different platforms. The default target compiles to a native binary for your current system. Cross-compilation to Windows and macOS is supported when the appropriate toolchains are installed.

# Current system (default)
python3 -m leash.cli compile program.lsh

# Windows 64-bit (cross-compile)
python3 -m leash.cli compile program.lsh --target win64

# macOS (cross-compile)
python3 -m leash.cli compile program.lsh --target macos

Library Output Types

Output TypeFlagOutput File
Executableto <name><name>
Dynamic Libraryto-dynamic <name>lib<name>.so
Static Libraryto-static <name>lib<name>.a
leash compile math.lsh to-dynamic math   # libmath.so
leash compile utils.lsh to-static utils  # libutils.a

Native targets produce standalone binaries with full access to the system's C library, garbage collector, and OS APIs.

Static Analysis

Leash provides thorough static analysis to catch errors and potential issues before your code runs. You can analyze a file without compiling it using the check command, which runs the full type checker with enhanced safety analysis.

The check Command

python3 -m leash.cli check program.lsh

The checker reports:

  • Type errors, undefined variables, unknown types
  • Duplicate switch cases
  • Shadowed globals
  • Missing switch defaults
  • Large stack arrays (>10,000 elements)
  • Self-assignments (x = x)
  • Unused variables and parameters
  • Empty blocks
  • Unreachable code
  • Always-true/false conditions
  • Redundant operations (+ 0, * 1, x == x)

The --check Flag

python3 -m leash.cli compile program.lsh --check
python3 -m leash.cli run program.lsh --check

Warnings as Errors

python3 -m leash.cli compile program.lsh --warnings-as-errors

Verbose Diagnostics

python3 -m leash.cli check program.lsh --verbose
python3 -m leash.cli compile program.lsh -vb

For strict builds (e.g., CI/CD), use the --warnings-as-errors flag to treat all warnings as errors, ensuring your codebase stays clean.

Verbose Diagnostics (--verbose / -vb)

For in-depth educational guidance, the --verbose or -vb flag prompts the compiler to print highly detailed, step-by-step masterclass explanations, diagnostics, and fully validated Leash code tips for any encountered errors or warnings. This is particularly useful when learning the language or debugging complex type issues.

Compiler Pipeline

Leash transforms source code into a native executable through a multi-stage pipeline. Each stage feeds its output into the next, from raw source text all the way to a linked binary. Here are the 15 stages:

#StageDescription
1LexerCharacters → tokens (keywords, operators, identifiers, literals)
2ParserTokens → AST (recursive-descent parser)
3Import ResolutionResolves use module::Item; statements
4Conditional CompilationEvaluates top-level if _PLATFORM == ...
5Macro ExpansionExpands defmacro definitions in-place
6Type CheckerValidates types, reports warnings
7Low-Level CheckerValidates memory layout and field invariants
8AST OptimizationConstant folding, DCE, dead branch elimination, TRO
9Code GenerationAST → LLVM IR in-memory module
10Alloca HoistingMoves all alloca to entry block
11Re-parse & VerifyParse IR and call verify()
12LLVM OptimizationLLVM pass pipeline (-O0 through -O4)
13Object EmissionLLVM IR → .o machine code
14Runtime CompilationCompiles gc.c + platform stubs
15LinkingLinks .o + GC + libraries → executable

Optimization Levels

Leash supports LLVM optimization levels via the --opt (or -O) flag, controlling the aggressiveness of the optimization pipeline across three layers: Leash AST (frontend), LLVM IR (middle-end), and C runtime (link-time). The -O flag controls all three simultaneously, so a single flag choice applies consistent optimization throughout the entire compilation process.

LevelDescriptionKey Passes
-O0No optimization (default)Fastest compilation
-O1BasicInstruction combining, DCE, CFG simplification
-O2StandardSROA, jump threading, reassociation, loop simplification
-O3AggressiveLoop unrolling, vectorization, inlining, IPSCCP, argument promotion
-O4MaximumTail recursion opt, aggressive inlining, dual-pass pipeline
-OsSizeO2 + global DCE + extra CFG simplification
-OzMinimum sizeAdditional size reductions
python3 -m leash.cli compile program.lsh --opt 3
python3 -m leash.cli run program.lsh -O s
Optimization occurs at three layers: Leash AST (frontend), LLVM IR (middle-end), and C runtime (link-time). The -O flag controls all three simultaneously.

Optimization Verbosity

python3 -m leash.cli compile program.lsh -ov
python3 -m leash.cli run program.lsh --opt 3 -ov

Displays detailed logs per phase:

  • [AST Opt] — AST-level pipeline phases
  • [CF] — Constant folding
  • [CP] — Constant propagation
  • [DBE] — Dead branch elimination
  • [UCE] — Unreachable code elimination
  • [DCE] — Dead code elimination
  • [TRO] — Tail recursion optimization
  • [LLVM Opt] — LLVM pass pipeline

The -ov flag is independent of --verbose/-vb — it only shows optimization-related messages, not masterclass diagnostics.

AST-Level Optimizations (Frontend)

Leash runs semantics-preserving AST passes before LLVM IR generation. These passes are always safe, never change program behavior, and are applied regardless of the optimization level selected. They form the foundation of Leash's optimization stack, catching simplification opportunities early in the compilation process.

PassEffect
Constant folding & propagationReplaces constant expressions with results at compile time
Dead branch eliminationRemoves if/else with constant-false conditions
Unreachable code eliminationRemoves code after return, exit(), throw
Dead code eliminationRemoves unused top-level functions, globals, types
Foreach small-loop unrollingAnnotates arrays ≤8 elements for LLVM unroller
Pushb fusion (future)Detects consecutive .pushb() for batch insertion
Size call cachingHoists .size() calls used 3+ times on same collection
Tail recursion optimizationConverts tail-recursive calls to jmp (-O4)

LLVM Codegen Optimizations

At the codegen layer, Leash emits optimal LLVM IR directly, setting flags and metadata that enable LLVM's own optimization passes to work more effectively. These are not separate optimization passes but rather smarter code generation that gives LLVM better information to work with.

OptimizationDetails
nuw/nsw flagsIndex arithmetic emitted with no-wrap flags for better LLVM folding
fast flagFloat operations use fast semantics (reassociation, reciprocal-math)
Branch weight metadataForeach back-edges weighted 99:1 for better register allocation
GC-tracked allocationVector buffers use leash_gc_malloc for reachability

Link-Time Optimization (LTO)

LTO enables whole-program analysis after linking, enabling optimizations across module boundaries. This is particularly powerful for projects that span multiple source files, as it allows the compiler to analyze and optimize across what were previously separate compilation units.

Benefits

  • Dead function elimination across modules
  • Cross-module inlining (e.g., library functions)
  • Devirtualization of virtual calls when the target is known
  • Constant propagation across translation units
# ThinLTO — best balance of speed and optimization
python3 -m leash.cli compile program.lsh --opt 2 --lto thin

# Full LTO — maximum cross-module optimization
python3 -m leash.cli compile program.lsh --opt 3 --lto full

Profile-Guided Optimization (PGO)

PGO uses runtime profiling data to optimize hot paths in your program. The workflow is a three-step process: first compile with instrumentation, run the program to collect real execution data, then recompile using that data to make better optimization decisions. This enables optimizations that would be impossible with purely static analysis.

# Step 1: Compile with instrumentation
python3 -m leash.cli compile program.lsh --opt 2 --pgo-generate

# Step 2: Run to collect profile data
./program

# Step 3: Recompile with profile data
python3 -m leash.cli compile program.lsh --opt 3 --pgo-use

PGO enables hot/cold code splitting, accurate branch prediction, and informed inlining and unrolling decisions based on actual execution frequency.

PGO enables:

  • Hot/cold code splitting — frequently-executed code is packed together for better cache locality
  • Accurate branch prediction — branches weighted by actual execution frequency
  • Inlining decisions — functions in hot paths are inlined first
  • Loop unrolling — unroll only hot loops based on actual trip counts

Runtime Safety Checks

Leash embeds runtime safety checks into compiled programs to catch common issues before they cause undefined behavior. All checks produce descriptive error messages with unique error codes (e.g., LEASH-E004) for easy reference. These checks can be disabled per-function using the unsafe keyword when maximum performance is needed.

CheckDescription
Division by zeroCaught at runtime with descriptive error message
Vector bounds checkingget(), set(), popb(), popf() validate indices and empty vectors
Null file handle checksFile operations on unopened/closed files produce clear errors
Union tag mismatchAccessing the wrong union variant halts with a runtime error

Conditional Compilation

Leash supports conditional compilation using if statements at the top level (outside any function). When the compiler encounters a top-level if, it evaluates the condition at compile time. Only the branch whose condition evaluates to true is included in the final program; the other branches are completely ignored (no code is generated, and they are not type-checked).

This is similar to C/C++ preprocessor #if but happens at the AST level, making it more integrated with the language. Conditions can check the target platform, file path, or file name, and can be combined with &&, ||, and ! operators.

if _PLATFORM == "linux64" {
    fnc main() : void {
        show("Running on Linux");
    }
} also _PLATFORM == "win64" {
    fnc main() : void {
        show("Running on Windows");
    }
}
