Concurrency Model

Leash supports multi-threaded programming through workers — special functions that run concurrently in separate OS threads. Workers communicate through two kinds of global variables: shared and fusion.

Shared Variables (shared)

A shared variable allows one thread to write while any number of threads read. This is the default safe pattern for producer-consumer scenarios where a single worker produces data and multiple workers consume it. Since only one writer is allowed, there's no risk of write-write conflicts, making shared the simplest and safest choice for one-to-many communication patterns.

shared result: int = 0;  // Only one writer, many readers
PropertyDescription
WritersOnly one worker may write at a time
ReadersAny number of workers may read
AtomicityNo guarantees — readers may see stale values

Fusion Variables (fusion)

A fusion variable allows multiple threads to both read and write concurrently. Writes use atomic stores and reads use atomic loads, so every thread eventually sees the latest value. This makes fusion the right choice for counters, flags, and coordination variables that many threads need to share and update.

fusion counter: int = 0;  // Multiple readers and writers
PropertyDescription
Readers & WritersAny worker may read or write
VisibilityChanges eventually visible to all threads (atomic semantics)
Use casesCounters, flags, values shared by many threads

Worker Functions (worker fnc)

Mark a function with worker fnc instead of fnc to make it runnable in its own OS thread. Workers are declared just like regular functions but with the worker keyword, accept parameters normally, and typically run in an infinite loop — checking thisworker.interrupted to know when the program is shutting down.

worker fnc calculate(a int, b int) {
    while !thisworker.interrupted {
        result = a + b + result;
        show("calculate updated result to: ", result);
    }
}
  • Workers are declared like regular functions but with the worker keyword
  • They accept parameters like normal functions
  • They run in an infinite loop by default
  • Use thisworker.interrupted to check if the program is shutting down

Spawning Workers (spawn)

Use spawn to launch a worker function in a new OS thread. Each spawn call creates a dedicated thread, passes arguments by value (copied into a heap-allocated struct), and starts execution immediately. Spawned threads run concurrently with main and with each other.

fnc main() {
    spawn calculate(3, 5);
    spawn increment_counter();
    spawn show_result();
}
  • Each spawn creates a new OS thread running the given worker function
  • Arguments are passed by value (copied into a heap-allocated struct)
  • All spawned threads run concurrently with main and with each other
  • Workers stop when the program terminates or when thisworker.interrupted becomes true (e.g., on Ctrl+C)

Full Example

shared result: int = 0;
fusion counter: int = 0;

worker fnc calculate(a int, b int) {
    while !thisworker.interrupted {
        result = a + b + result;
        show("calculate updated result to: ", result);
    }
}

worker fnc increment_counter() {
    while !thisworker.interrupted {
        counter = counter + 1;
        show("counter incremented to: ", counter);
    }
}

worker fnc show_result() {
    while !thisworker.interrupted {
        show("result:", result, " counter:", counter);
    }
}

fnc main() {
    spawn calculate(3, 5);
    spawn increment_counter();
    spawn show_result();
}

Lifecycle & Cleanup

  • Workers are automatically started when spawn is called
  • They run until the program exits or thisworker.interrupted is true
  • Ctrl+C sets the interrupted flag for clean shutdown
  • The runtime waits for all workers to finish before full exit
  • Memory allocated in workers is handled by the thread-safe garbage collector
