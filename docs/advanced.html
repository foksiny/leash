Advanced Topics

Memory Management

Leash uses a custom garbage collector with a mark-and-sweep algorithm and conservative root finding. Memory is managed automatically — you never need to call free() or worry about memory leaks.

How It Works

  1. Mark Phase: Starting from root pointers (globals, stack variables), the GC traces all reachable objects
  2. Sweep Phase: Any object not marked as reachable is freed automatically

Features

  • Conservative root finding — identifies pointers without explicit registration
  • Explicit root management via leash_gc_register_root() / leash_gc_unregister_root()
  • Automatic collection when memory exceeds a threshold
  • Type-aware: strings, vectors, arrays, structs, classes, unions all handled properly
  • Built-in GC stats via leash_gc_print_stats()

Memory Lifecycle

The GC is automatically initialized when your program starts, and all memory allocation in Leash goes through the custom GC (leash_gc_malloc, leash_gc_realloc). The GC runtime is compiled and linked with every Leash program automatically — no external GC library (like Boehm) is needed. When variables go out of scope or become unreachable, memory is automatically reclaimed on the next GC cycle.

fnc main() : void {
    // Strings are GC-managed
    name: string = "Leash";

    // Vectors grow dynamically, GC handles reallocation
    v: vec<int>;
    for i: int = 0; i < 1000; i = i + 1 {
        v.pushb(i);
    }
    // Memory is automatically reclaimed when variables go out of scope
}

Manual Memory Management (nogc)

For low-level systems programming, Leash supports manual memory management via the nogc function modifier. Mark a function with nogc to use C's malloc/free instead of the GC allocator:

unsafe nogc fnc main : int {
    buf: *char = (*char)malloc(256);
    strcpy(buf, "Hello, raw heap!");
    printf("%s\n", buf);
    free(buf);
    return 0;
}

Key Behaviors

  • Memory uses the C heap directly
  • GC is not initialized for nogc main (saves startup time)
  • Calling nogc from GC-managed code produces a warning
  • Use unsafe nogc together to suppress safety checks

Global --no-garbage-collector / -ngc Flag

leash run myprogram.lsh --no-garbage-collector
leash compile myprogram.lsh -ngc
-ngc is intended for embedded, real-time, or minimal-binary-size scenarios. Use with extreme care.

Error Handling & Safety

Leash prioritises developer experience with helpful error reporting and safety features including static type checking, smart error tips, runtime union checks, and unique error codes for every warning.

Custom Error Definitions

Leash allows you to define custom errors with the error keyword. Errors can take arguments and have a message expression that is evaluated when the error is thrown. When an error is thrown, the message is printed along with file, line, and column info, and the program terminates with exit code 1 (unless caught in a works block).

error ExpectedArguments(amount int, got int) -> "Expected " + tostring(amount) + " or more arguments but only got " + tostring(got) + ".";

fnc main(args string[]) {
    if args.size < 2 {
        throw ExpectedArguments(2, args.size);
    }
}

When an error is thrown, the message is printed with file, line, and column info, and the program terminates with exit code 1.

Works-Otherwise Error Handling

Leash provides a works...otherwise construct for catching and handling errors. Unlike traditional try-catch, it can catch any error — undefined variables, type errors, runtime errors, and more — allowing the program to recover and continue gracefully.

fnc main() : void {
    works {
        a: int = unknown_var;  // normally an error
    } otherwise err {
        show("Error caught: ", err);
    }
    show("Program continues...");
}

How it works: The works block attempts to execute its statements. If any error occurs, instead of crashing, control jumps to the otherwise block where the error variable contains a string describing what went wrong. After the otherwise block completes, the program continues normally.

If no error occurs, the otherwise block is skipped entirely. This makes works...otherwise useful for graceful error handling, fallback logic, and recovering from unexpected conditions.

Unsafe Functions

The unsafe keyword disables runtime safety checks and static safety warnings within a function. This is useful for low-level operations like bit manipulation, type punning through unions, or performance-critical code where you want to bypass the overhead of safety checks. unsafe also works on class methods.

unsafe fnc fabs(x float<64>) : float<64> {
    mem: Memory = x;
    mem.i = mem.i & 0x7FFFFFFFFFFFFFFF;  // Clear sign bit
    return mem.f;
}

Safety checks disabled by unsafe:

  • Division by zero runtime check
  • Union tag mismatch detection
  • Null pointer dereference checks
  • Vector bounds checking
  • Static division-by-zero warning
  • Static array bounds checks
Use unsafe only when you truly need to bypass safety checks. Unsafe code can cause undefined behavior, crashes, or memory corruption.

Native Library Imports (FFI)

Leash supports importing functions from native libraries (.a or .so files) using the @from directive. This enables Foreign Function Interface (FFI) with C, C++, Rust, and other languages that compile to static or shared libraries. The @from directive tells the compiler to declare items as external and link against the specified library during compilation.

@from("mylib.so") {
    fnc add(a int, b int) : int;
    fnc multiply(a int, b int) : int;
    my_global: int;

    def Point : struct {
        x: int;
        y: int;
    };

    def Color : enum { RED, GREEN, BLUE };
    def MyInt : type int;
};

fnc main() : void {
    show(add(10, 20));  // 30
}

Creating Native Libraries for Leash

// math_utils.c
int add(int a, int b) {
    return a + b;
}
# Compile to static library
gcc -c math_utils.c -o math_utils.o
ar rcs libmath.a math_utils.o

# Or compile to shared library
gcc -shared -fPIC math_utils.c -o libmath.so

Limitations

  • Type signatures must match exactly between Leash and the native library
  • The library must be available at link time
  • Initializers are not supported for variable declarations
  • Only basic struct/union/enum fields supported

Program Termination

exit(1);  // terminates immediately with exit code

C Runtime Optimizations

The Leash runtime (gc.c / gc.h) is compiled at -O3 and includes:

#Optimization
1Function-pointer matrix dispatch — single indirect call per element-wise op instead of switch
24× loop unrolling with __builtin_prefetch
3Reusable thread pool — replaces per-call thread creation (lazily initialized, persists for program lifetime)
4Cache-blocked operations — processes tiles fitting in L1 cache
5Vector batch utilities — bulk copy, reverse, sort (O(n log n))
6Thread-local allocation (TLAB) — bump-pointer arena per thread
7GC bitmap — compact bitmask for faster mark phase
8Fast memcpy — thin wrapper with __restrict__ hints

Syntax Highlighting

Leash provides native syntax highlighting for popular editors. Files are in the syntax_highlighters/ directory.

Vim / Neovim

mkdir -p ~/.vim/syntax
cp syntax_highlighters/leash.vim ~/.vim/syntax/
# Add to .vimrc:
" au BufRead,BufNewFile *.lsh set filetype=leash

VS Code (with LSP Support)

The VS Code extension provides syntax highlighting, real-time diagnostics, hover tooltips, go-to-definition, and auto-completion.

Manual installation:

# Copy to VS Code extensions folder
# Windows: %USERPROFILE%\.vscode\extensions\leash
# macOS/Linux: ~/.vscode/extensions/leash

# Or build from source
cd syntax_highlighters/vscode
npm install
npm run compile

Emacs

mkdir -p ~/.emacs.d/lisp
cp syntax_highlighters/leash-mode.el ~/.emacs.d/lisp/
# Add to init.el:
;; (add-to-list 'load-path "~/.emacs.d/lisp/")
;; (require 'leash-mode)
